Prompt 與脈絡完整指南

好消息：寫好 Prompt 需要的能力——釐清目標、給足脈絡、定義驗收標準——正是 PM 每天在做的事。 你不是在學一項新技能，是在把「寫需求」的能力用在新對象上。

核心觀念：Prompt 是給 AI 的需求單

想像你把任務交給一位第一天到職、能力很強但完全不了解公司的新同事。你會怎麼交代？

  • 你不會只說「幫我寫個 PRD」——你會說清楚是什麼功能、給誰用、解決什麼問題。
  • 你不會假設他知道你們的用戶是誰——你會給他背景資料。
  • 你不會不講交付格式——你會說「用我們的 PRD 模板，兩頁以內」。

對 Claude 也一樣。模糊的輸入，只會得到通用的輸出。

四元素拆解（含 PM 實例）

1. 角色 Role — 設定視角與專業水準

你是一位有 10 年經驗的 B2B SaaS 資深產品經理，擅長把模糊的商業需求轉成可執行的規格。

2. 任務 Task — 動詞明確、範圍清楚、有驗收標準

請閱讀以下 5 份用戶訪談摘要，萃取出前 5 大痛點，按「提及人數」排序，每個痛點附上一句最有代表性的原話。

3. 脈絡 Context — 產出品質的天花板

背景：我們是一個給中小企業用的排班 SaaS，主要用戶是餐飲業店長，付費方案月費 $990。
這次訪談的目的是驗證「員工臨時請假造成排班混亂」是否為付費升級的關鍵動機。
已知限制：本季工程資源只剩 3 人週，方案必須輕量。

4. 格式 Format — 決定產出可不可以直接用

輸出格式：
- 用表格呈現：痛點 | 提及人數 | 代表原話 | 對付費動機的影響（高/中/低）
- 表格後面用 3 句話總結「對本季 roadmap 的建議」
- 繁體中文，總長度不超過 400 字

進階技巧：讓產出再上一層

  • 給範例（Few-shot）：附上一份「你覺得寫得好的 PRD」，說「照這個水準和結構寫」。一個好範例勝過十行形容詞。
  • 要求先思考再回答：「先列出你的分析步驟，確認後再執行」——複雜任務的品質會明顯提升，你也能在方向錯誤時及早修正。
  • 指定不確定時的行為：「資料不足以下結論時，直接說『證據不足』，不要猜」——這是對抗幻覺最有效的一句話。
  • 反向提問：「開始之前，先問我 3 個能讓產出更精準的問題」——讓 AI 幫你發現自己漏給的脈絡。
  • 迭代而非重來：第一版產出後，用具體的修改指令收斂：「第二段太籠統，改成引用實際數據」比「寫好一點」有效得多。

對照：同一個任務的兩種問法

❌ 模糊版✓ 專業版
「幫我寫一份深色模式的 PRD」 角色：資深 PM。任務：為行動端 App 撰寫深色模式 PRD 初稿。脈絡：用戶調查中 34% 主動提及、競品 A/B 均已支援、設計系統已有色彩 token、本季只能投入 2 人月。格式：含問題陳述、目標與非目標、成功指標、分階段範圍，兩頁內。

模糊版會得到一份「教科書式」的通用 PRD；專業版會得到一份可以直接進評審會修改的初稿。 差別不在 AI，在輸入。
