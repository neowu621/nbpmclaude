Skill（技能包）把重複工作打包成可重用能力

你已經學會寫好的 Prompt（第 4 章）、也知道重複的工作該自動化（第 7 章）。 Skill 就是這兩件事的交會點：把一個驗證過的工作流程固化成可重用的能力， 之後一句話就能叫出來，不用每次重寫 Prompt。這頁講清楚：Skill 是什麼、什麼時候該用、怎麼建、 以及 PM 最該優先建立的 8 個 Skill。

✓ 一句話理解：Prompt 是「這次怎麼做」，Skill 是「以後每次都這樣做」—— 它是一份給 AI 的工作手冊：預先寫好步驟、輸出格式與品質標準。
SECTION 01

Skill 是什麼？跟 GPTs、Projects 差在哪？

引導問題：都是「讓 AI 記住我要什麼」，為什麼要分成三種東西？

OpenAI 對 Skill 的定義是：一份可重用的指令集，教 AI 依標準流程執行特定任務—— 裡面預先寫好步驟、輸出格式與品質標準，像一本工作手冊。 關鍵差別在「它解決的是哪一種重複」：

類型本質最適合
Skill重複任務的標準作業流程週報、會議紀錄、固定格式的產出
Custom GPTs有特定領域與人格的AI 助理長期、專業固定的顧問角色
Projects集中脈絡的協作工作區持續進行的專案，如產品上線

用 PM 的話說：Projects 是「共用的資料夾與脈絡」，GPTs 是「一位固定角色的同事」，Skill 是「一份 SOP」。 三者不衝突——你可以在一個 Project 裡，用一個 Skill 產出這週的週報。

✓ 對照第 3 章：Claude 這邊的對應是 Cowork 模式的 Projects（放入文件與自訂指令，讓每次對話自帶脈絡）；Claude 也有 Skills 機制。概念是共通的：把脈絡與流程固化，不要每次重講。
SECTION 02

什麼情況、什麼場景適合做成 Skill？

引導問題：不是所有事都值得做成 Skill——那界線在哪？

沿用第 7 章的判斷公式：頻率 × 每次耗時 × 格式穩定度。三個條件同時成立，才值得固化：

  1. 會重複發生：每週／每月都要做一次（一次性的事don't）。
  2. 流程已經穩定：你已經手動跑順至少三次，知道好的產出長什麼樣。流程還在變的東西，固化下來只會綁住自己。
  3. 輸出格式固定：每次的結構一樣，只有內容不同（週報、PRD、會議紀錄都符合）。
⚠️ 不適合做成 Skill 的：需要大量判斷與取捨的事（優先級的最終拍板、對高層的承諾、產品願景）。 這些落在第 2 章光譜的「判斷」端——Skill 能標準化的是「產出」，不是「判斷」。 把判斷做成 Skill，等於把你的價值自動化掉。
SECTION 03

三種方式建立自己的 Skill

以 ChatGPT Skills 為例（合格帳號預設就內建一個 skill-creator 可以幫你做）：

方式怎麼做適合誰
① 對話式開新對話直接說「幫我建一個 skill…」，它會反問你流程、輸出格式與需要的工具，然後直接存成 Skill新手首選——不用碰任何格式
② 編輯器到 chatgpt.com/skills → 「+」→ build via editor，直接用 Markdown 寫 SKILL.md，可附加資源檔想精確控制步驟與格式
③ 檔案上傳到 chatgpt.com/skills → 「+」→ upload from computer，上傳寫好的 SKILL.md；系統會做安全掃描，結果分為可用／需人工確認／封鎖沿用團隊既有的 SOP 文件

PM 的建議做法：先用①對話式做出來，跑幾次覺得穩了，再用②編輯器微調步驟與格式。不要一開始就想寫完美的 SKILL.md。

怎麼呼叫？兩種情境

  • @ 指定（手動）：在輸入框打 @，從清單選 Skill。例：@weekly-report 把這段文字整理成進度更新。你確定要用哪個時最快。
  • 自動觸發：ChatGPT 依 Skill 的名稱與描述自行判斷要不要用，複雜任務甚至會組合多個 Skill。
    關鍵：描述寫得越詳細，自動觸發越準——這正是六條寫法原則的「指定得夠清楚」在 Skill 層級的應用。
SECTION 04

官方點名的 6 大職場 Skills

OpenAI 官方示範的六個職場 Skill——注意它們全都是「把散亂輸入變成固定格式產出」，沒有一個在做判斷：

領域Skill功能
行銷campaign-brief-builder把初步想法變成結構化的行銷 brief
業務discovery-to-next-steps從會議筆記萃取痛點與後續行動
財務monthly-close-narrative自動生成財報說明與差異分析
營運ops-playbook-writer把日常流程寫成文件化 SOP
管理weekly-status-multi-format把團隊更新彙整成三種格式
高階主管exec-decision-brief把研究濃縮成一頁決策摘要
SECTION 05

⭐ PM 該優先建立的 8 個基本 Skill

引導問題：如果只能先做三個，你會選哪三個？（提示：挑你每週都在做、格式又固定的）

依「頻率 × 耗時 × 格式穩定」排序，這八個是 PM 投報率最高的。每個都對應手冊裡已經驗證過的任務——先去做那個任務、跑順三次，再固化成 Skill：

1weekly-status
週報彙整——最該第一個做
輸入：散落的進度筆記、指標數字 · 輸出：固定格式週報（進展／風險／下週重點）
→ 每週一次、格式最穩定；先做 T09 週報自動化
2meeting-to-actions
會議紀錄轉行動項目
輸入：逐字稿或速記 · 輸出：決議 / 待辦（負責人＋期限）／ 未解問題
→ 高頻、高價值；先做 T10 會議紀錄
3interview-extractor
訪談逐字稿萃取痛點
輸入：多份逐字稿（去識別化）· 輸出：痛點主題＋提及頻率＋代表引述
→ 每輪研究都要做；先做 T05 逐字稿萃取
4prd-drafter
依團隊模板產出 PRD 骨架
輸入：問題描述、目標用戶、限制 · 輸出：PRD（含目標與非目標、成功指標、分階段範圍）
→ 把「你們團隊的 PRD 長什麼樣」固化進去;先做 T02 PRD 初稿
5exec-brief
長文／研究 → 一頁決策摘要
輸入：研究報告、數據、多方意見 · 輸出：一頁：結論／依據／選項／建議決定
→ 對應官方 exec-decision-brief；向上溝通的固定武器
6spec-stress-test
規格送審前的四角色壓力測試
輸入：你的 PRD · 輸出：工程／設計／客服／法務各 3 個尖銳問題＋建議答案
→ 每次送審都跑一次；先做 T03 規格壓力測試
7rice-ranker
需求清單 → RICE 評分與排序
輸入：需求清單與已知資訊 · 輸出：RICE 評分表＋排序＋每項的理由與不確定性
→ ⚠️ 產出評分是 Skill 的事，最終排序拍板是你的事；先做 T14 RICE 排序
8competitor-digest
競品動態定期彙整
輸入：競品官網／發布說明／新聞 · 輸出：變化摘要＋對我方的意涵＋建議關注點
→ ⚠️ 事實必查證（第 1 章幻覺）；先做 T13 競品分析
✓ 只做三個的話：weekly-status → meeting-to-actions → prd-drafter。這三個涵蓋 PM 最高頻的重複產出，做完你每週至少省下半天。
SECTION 06

使用前必看：3 大限制與安全規範

  1. 不會跨平台同步：在 ChatGPT 建的 Skill 不會自動出現在 Codex 或其他產品，要手動下載再上傳。
  2. 外部來源有安全風險：上傳的 Skill 會經過安全掃描（結果分「可用／需人工確認／封鎖」）。來路不明的 Skill 不要裝——它可能導致資料外洩。
  3. 企業有權限控管：商用／教育環境中，管理員可決定使用者能不能建立、上傳、分享或安裝 Skill，依你的工作區角色而定。
⚠️ 帳號限制（截至本文整理時）：ChatGPT Skills 仍在 Beta——目前開放 Business、Enterprise、Edu、Teachers、Healthcare 與 Codex／API； Plus 與 Pro 即將支援；免費版與 Go 尚未支援。
也就是說：多數個人帳號現在還用不到——但概念與做法先學起來，等開放時直接上手。 而且把流程寫成 SOP 這件事，本身就有價值（見第 8 章與協作工具包）。
✓ 呼應第 8 章：Skill 裡若寫入公司流程與範例，等於把內部資料放進 AI——先確認你們的 AI 使用政策與資料分級再建。
SECTION 07

進階應用與下一步

官方與實測案例顯示，Skills 最有感的場景是日常行政的重複工作——例如報帳流程、資料分類：輸入格式固定、規則明確、每次都在做同一件事。這正好符合前面的三個條件。

你的行動順序

  1. 翻開任務實作，挑一個你每週都在做的任務（建議 T09 週報或 T10 會議紀錄）。
  2. 用任務裡的 Prompt 手動跑順三次，每次微調到產出你滿意。
  3. 把那個驗證過的 Prompt 存進團隊 Prompt 庫（自動化階梯第一階）。
  4. 等 Skills 開放你的帳號，用對話式把它變成 Skill（「幫我建一個 skill…」）。
  5. Claude 這邊的等效做法：建一個 Project 放入文件與自訂指令，讓每次對話自帶脈絡。
✓ 核心原則不變：先手動跑順三次，再自動化。流程還在變的東西不要急著固定——這跟你對開發團隊說「先驗證再規模化」是同一個道理。
