Skill（技能包）把重複工作打包成可重用能力

⏸ 先確認：你現在需要讀這頁嗎？

這是進階頁。如果你還沒做過至少 3 個任務實作， 你現在還不需要 Skill——直接複製現成 Prompt 就好，那樣完全沒問題，很多資深 PM 一輩子都停在這裡也用得很好。

Skill 要解決的是一個很具體的痛：「同一段 Prompt 我這個月已經貼第 12 次了」。 還沒有痛到這個程度，就先別花時間。把這頁存書籤，等你真的貼煩了再回來。

你已經學會寫好的 Prompt（第 4 章）、也知道重複的工作該自動化（第 7 章）。 Skill 就是這兩件事的交會點：把一個驗證過的工作流程固化成可重用的能力， 之後一句話就能叫出來，不用每次重寫 Prompt。這頁講清楚：Skill 是什麼、 Claude / ChatGPT / Gemini 三家怎麼做、什麼時候該用、怎麼建、 以及 PM 最該優先建立的 8 個 Skill。

✓ 一句話理解：Prompt 是「這次怎麼做」，Skill 是「以後每次都這樣做」—— 它是一份給 AI 的工作手冊：預先寫好步驟、輸出格式與品質標準。
SECTION 01

Skill 是什麼？用「新人入職」來理解

引導問題：為什麼「教一次，以後都會」這件事，需要一個專門的機制？

想像你帶一位很聰明但完全不認識你們公司的新人。他什麼都學得會，但有個怪毛病： 每天早上都會忘記昨天教過的所有事。

你有三種應對方式，這三種剛好就對應到 AI 的三個機制：

你的做法對應機制白話
每次都重講一遍該怎麼做 Prompt 這次怎麼做。講完就沒了，下次再講一次。
把公司資料放在他桌上的共用資料夾 Projects（Claude）
Projects（ChatGPT）		 他要用的材料。但不會告訴他「怎麼做」。
寫一份 SOP 貼在牆上：「做週報時，照這 5 步」 Skill 以後每次都這樣做。他自己會判斷什麼時候該翻這份 SOP。
✓ 關鍵差別：Projects 給的是材料，Skill 給的是做法。 兩者常常一起用：Skill 說「照這 5 步寫週報」，Projects 提供「這週的 Jira 匯出檔」。
SECTION 02

Claude / ChatGPT / Gemini 三家怎麼做？

引導問題：我學的東西，換一個 AI 工具還能用嗎？

這是最多人問、也最多人搞錯的一題。答案是：一半通用，一半不通用，而且分界線很清楚。

✅ 好消息：SKILL.md 已經是開放標準

2025 年 12 月起，SKILL.md 這個格式變成跨廠商的開放標準， 目前約 30 多個 AI 工具都讀得懂——包含 Claude Code、Gemini CLI、Codex CLI、Cursor 等。 同一份 SKILL.md 檔案，不用改，三家都能用。 你在這頁學的 Skill 寫法，不會因為換工具而作廢。

但這個「通用」有個重要前提：它通用在開發者工具（CLI）那一層。 如果你用的是網頁版 App（claude.ai / chatgpt.com / gemini.google.com），三家的機制、名字、限制都不一樣：

✓ 給 PM 的實務結論： 概念 100% 通用——「把重複工作寫成 SOP 給 AI」這件事三家都成立，你學的思路不會浪費。 但 GPTs 和 Gems 不是 Skill：它們比較像「一位設定好人設的同事」（≈ 客製助理）， 而 Skill 是「一份 SOP」——AI 自己判斷何時該翻開它。這是本質差異，不是名字差異。
⚠️ 這一塊變動非常快

上表是 2026 年中的狀況，而且幾乎確定會再變（例：Gemini CLI 已宣布 2026/6/18 對個人用戶終止， 由 Antigravity CLI 接手，Skills 機制保留但資料夾從 .gemini/skills/ 改到 .agents/skills/）。 你該記住的是概念，不是這張表。要動手前，請去 官方資源確認當下的實際規格。

SECTION 03

什麼時候該做成 Skill？（三個條件全中才做）

引導問題：我怎麼知道這件事值不值得花 30 分鐘固化？

做 Skill 有成本。這三個條件全部成立再動手，否則你只是在做一份沒人用的文件：

條件怎麼判斷反例（別做）
1. 夠常做 一個月至少 4 次以上 一年一次的年度規劃 → 直接寫 Prompt
2. 流程穩定 步驟每次都差不多，你已經驗證過至少 3 次 還在摸索怎麼做 → 先用 Prompt 迭代到穩定
3. 產出格式固定 你能明確說出「好的產出長什麼樣」 「幫我想點子」→ 沒有固定格式，不適合
✓ 一個很實用的檢查法： 翻你自己的對話紀錄，看有沒有同一段 Prompt 你貼了 3 次以上。 有 → 那就是你的第一個 Skill，而且題目已經幫你選好了。 沒有 → 你還不需要 Skill，回去做任務。
SECTION 04

一個 Skill 長什麼樣子？（看真的檔案）

引導問題：不會寫程式的 PM，做得出 Skill 嗎？

先講結論：做得出來，而且不用寫任何程式。 一個 Skill 最小的樣子，就是一個資料夾，裡面放一個叫 SKILL.md 的純文字檔。 .md 是 Markdown，就是「可以用 # 標題、- 列點的純文字」，你在寫 Notion 的時候其實就在用它。

這是一個真實可用的 Skill，整份就這麼長：

--- ← 這三個橫線之間叫「frontmatter」，是給 AI 看的標籤 name: weekly-status description: 產生產品週報。當使用者要求「寫週報」、「本週進度」 或提供 Jira／Linear 匯出資料要求彙整時使用。 --- # 產品週報產生器 ## 步驟 1. 讀取使用者提供的 Jira／Linear 匯出資料 2. 依「已完成／進行中／受阻」三類分組 3. 受阻項目必須標出卡住的原因與需要誰決策 4. 資料不足的欄位標【待確認】，不要猜測 5. 「風險判讀與建議」段落留空——那是 PM 的工作，不是你的 ## 輸出格式 - 總長度不超過 400 字 - 開頭一句話講「這週最重要的一件事」 - 給主管看，不要有內部術語 ## 品質標準 - 每個「已完成」都要對得上一個 ticket 編號 - 不確定的事寧可標【待確認】，也不要寫得很篤定

整份檔案只有兩個地方是必填的，其他都是你自己的 SOP 內容：

欄位作用寫壞會怎樣
name 這個 Skill 的代號 叫不出來
description 最關鍵的一行。AI 靠它判斷「什麼時候該用這個 Skill」 寫太模糊 → AI 該用的時候不用，或不該用的時候亂用
✓ 90% 的 Skill 失敗都敗在 description： 寫「處理週報相關事務」→ AI 不知道什麼算「相關」，常常不觸發。 寫「當使用者要求『寫週報』、『本週進度』，或提供 Jira 匯出資料要求彙整時使用」→ 明確列出觸發時機與關鍵詞，AI 才叫得動。 把 description 想成「這份 SOP 的封面標題」——同事要能一眼看出什麼時候該翻它。

為什麼要分成 frontmatter 和內文？（漸進式揭露）

這個設計有個很聰明的理由，理解它你就懂 Skill 的整個邏輯了。AI 的「注意力」是有限資源（詞彙表：Context）， 不可能把你所有 Skill 的全文都一直放在腦子裡。所以它分三層載入：

三種建立方式

方式適合誰怎麼做
1. 叫 AI 幫你寫
最推薦		 所有人，尤其新手 把你貼過 3 次的那段 Prompt 丟給 AI：「幫我把這段整理成一個 SKILL.md，description 要明確寫出觸發時機」
2. 手寫 流程你自己最清楚時 照上面的範本改。真的就是純文字，沒有魔法。
3. 用現成的 通用任務（做投影片、Excel） 各家都有官方預建 Skill，直接開來用
✓ 新手就用第 1 種。你不需要學 Markdown 語法、不需要記 frontmatter 格式—— 叫 AI 寫，你負責檢查它寫得對不對。這正是本手冊的主軸： AI 產出，你判斷。
SECTION 05

⭐ PM 最該優先建立的 8 個 Skill

引導問題：如果只能做 3 個，我該做哪 3 個？

這 8 個是從任務實作裡挑出「最常重複、格式最固定」的。 不要一次做 8 個——挑你這個月真的做過 4 次以上的那 1～2 個開始。

1weekly-status
產品週報產生器
輸入：Jira／Linear 匯出　輸出：已完成／進行中／受阻三段式週報
💡 為什麼優先：每週都做，格式最固定，投報率最高。→ T09
2meeting-to-actions
會議紀錄轉行動項目
輸入：會議逐字稿／筆記　輸出：誰、做什麼、何時完成的行動清單
💡 為什麼優先：頻率最高的 PM 雜事，而且格式極度固定。→ T10
3interview-extractor
訪談逐字稿萃取
輸入：用戶訪談逐字稿　輸出：痛點／需求／原話引用
⚠️ 幻覺風險最高的一個——Skill 裡務必寫死「原話必須可回溯原文」。→ T05
4prd-drafter
PRD 初稿
輸入：需求描述　輸出：依你們公司範本的 PRD 骨架
💡 為什麼優先：能把「公司格式」固化，省下每次解釋範本的力氣。→ T02
5exec-brief
給主管的一頁摘要
輸入：任何長文件　輸出：結論先行、300 字內、無術語
💡 為什麼優先：「換一種聽眾講一次」是 PM 最常做的轉換。→ T11
6spec-stress-test
規格壓力測試
輸入：你的規格　輸出：邊界情境、遺漏、反問清單
💡 為什麼優先：這是少數「AI 比你想得周全」的場景，而且每次規格都該跑一次。→ T03
7rice-ranker
RICE 排序
輸入：需求清單　輸出：RICE 四維評分表＋計算過程
⚠️ 只固化「算法」，不要固化「分數」——分數是你的判斷。→ T14
8competitor-digest
競品分析摘要
輸入：競品資料／公開資訊　輸出：功能對照＋差異點
⚠️ Skill 裡務必寫死「模型既有知識必須標注需查證」——這題翻車過。→ 見實務案例
⚠️ 絕對不要做成 Skill 的東西：你的判斷

「該不該做這個功能」、「這個風險能不能接受」、「要不要延期」—— 這些不能固化，因為它們每次的答案都該不一樣。 如果你發現自己想把「決策」寫進 Skill，通常代表你正在把責任外包給一個不用負責的系統。

注意上面 8 個 Skill 的共同點：它們全都停在「整理好給你看」，沒有一個幫你決定。 weekly-status 把「風險判讀」留空、rice-ranker 只算不排——這是刻意的。 AI 指出，PM 決定。

SECTION 06

三個你會踩到的坑

坑症狀解法
Skill 叫不出來 你設好了，但 AI 完全不理它 9 成是 description 太模糊。改成明確寫出觸發關鍵詞。
Skill 亂觸發 你只是閒聊，它硬要幫你寫週報 description 範圍寫太寬。加上「僅在…時使用」。
Skill 過期了 流程改了，但 Skill 還在照舊做法產出 這是最危險的。Skill 會「錯得很穩定、很準時」。排一個每季檢查。
✓ 自動化的鐵律（第 7 章講過，這裡再講一次）： 自動化最大的風險不是「做錯」，是「錯了也每週準時錯」—— 而且因為它一直很穩定，沒有人會去檢查。 每個 Skill 都該有一個負責人和一個複查週期。
SECTION 07

你的下一步

  1. 先別建 Skill。去翻你這個月的 AI 對話紀錄。
  2. 找出貼了 3 次以上的同一段 Prompt。找不到 → 你還不需要 Skill，回去做任務。
  3. 找到了 → 把它丟給 AI：「幫我把這段整理成 SKILL.md，description 要明確寫出觸發時機與關鍵詞」。
  4. 連續用兩週，記錄它哪裡不準。
  5. 改一次 description，再用兩週。穩定了才做第二個。
✓ 自我檢視：你能說出一個「你這個月至少做過 4 次、步驟穩定、產出格式固定」的工作嗎？ 如果能——那就是你的第一個 Skill。如果不能——那你現在最該做的不是 Skill，是先累積夠多的重複。

延伸閱讀：Prompt Library（提示詞庫）—— Skill 的內容就是好 Prompt，那頁的六條寫法原則直接適用。
