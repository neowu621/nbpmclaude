Prompt Library（提示詞庫）讓 AI 更懂你

Anthropic 官方釋出了 Claude Code 的 Prompt Library——52 組現成提示詞，依開發生命週期的五個階段分類。 這頁不是要你背 52 組指令，而是帶你看懂它背後的六條寫法原則： 為什麼同一件事，有人問出垃圾、有人問出可以直接用的產出。這正是第 4 章「Prompt 是需求文件」的實戰升級版。

✓ 先講結論：這 52 組提示詞的價值不在「複製貼上」，而在它們共用同一套寫法—— 描述結果、內建驗證、給範式、給指標、給素材、指定格式。學會這六條，你自己就能寫出第 53 組。
SECTION 01

六條寫法原則：讓 AI 更懂你

引導問題：官方 52 組提示詞長得都不一樣，為什麼它們「有效」的理由是同一個？
1
描述結果，不描述步驟

別指定「去改 A 檔案第 30 行」，而是說「我要讓新用戶三步內完成設定」——讓 Claude 自己找出相關檔案與路徑。你負責定義「要什麼」，它負責找「在哪、怎麼做」。

PM 對照：這就是你寫需求單的方式——描述問題與目標，不是幫工程師寫實作。

2
內建自我驗證

在同一個提示詞裡同時要求「執行 + 驗證」，例如「寫完後跑測試，失敗就修到全過」。Claude 會自己迭代到通過，而不是丟一份沒驗證的產出給你。

PM 對照：等於在需求單裡就寫好驗收標準——沒有驗收標準的需求，交付品質全靠運氣。

3
提供參考範式

給它你既有的寫法／文件／範例，它會照你的風格做，而不是套用通用最佳實踐。「照 X 的格式做 Y」比「做一個好的 Y」有效得多。

PM 對照：附上上一版 PRD 當範本，勝過解釋十句「我想要的風格」。

4
用可量測的目標

把「優化效能」換成「首屏載入壓到 2 秒內」。有具體指標，才有明確的完成標準，AI 才知道什麼時候算做完。

PM 對照：呼應第 7 章的成功指標——沒有數字的目標，是願望不是需求。

5
給原始素材，不要給摘要

直接貼錯誤日誌、截圖、原始數據、計畫輸出——AI 處理原始資料的能力，遠勝你先幫它濃縮過的二手摘要（你的摘要可能已經漏掉關鍵線索）。

PM 對照：第 4 章「脈絡的品質決定產出的上限」的具體做法。

6
指定輸出格式

說清楚格式、長度、受眾：「輸出成表格，200 字內，給不懂技術的高階主管看」。不說，就會拿到一份無法直接用的東西。

PM 對照：第 4 章四元素中的「格式」——決定產出能不能直接貼進你的文件。

✓ 自我檢核：拿你最近一個提示詞出來，對照這六條——你最常漏掉的是哪一條？（多數 PM 漏的是第 2 條「自我驗證」與第 4 條「可量測目標」。）
SECTION 02

五大階段：52 組提示詞的分類邏輯

引導問題：這個庫依「開發生命週期」分類——但哪些階段其實跟 PM 有關？

官方把提示詞依開發流程的先後順序分成五個階段。以下標注哪些對 PM／設計師特別有用：

① Discover — 探索與理解

🎯 PM／設計師高度相關
  • 看懂不熟的程式碼：不用指定檔案，直接問「這個系統的架構長怎樣」。
  • 找功能在哪、追程式碼演進history。
  • 刪除前的影響分析：「拿掉這個功能會影響什麼？」——評估改動範圍的利器。
  • 產品層級的行為問題：用產品語言問，不用工程語言。

② Design — 規劃與原型

🎯 PM／設計師高度相關
  • 先列計畫、不要動手：把「探索」與「執行」分開——Claude 先列出要改什麼，你確認後才做。
  • 訪談式寫規格：讓它反問你，逼出你沒想到的邊界情境。
  • 會議記錄轉工作項目、找出錯誤／空狀態（設計常漏的地方）。
  • mockup 轉可點原型、設計與實作的自動比對。

③ Build — 實作、測試、重構、審查

最大的一類，多數偏工程；但有幾組 PM 用得到
  • 給非工程師的內部小工具、依範本產出文件草稿、批次文字更新。
  • 發布前的內容審查（行銷／文件團隊適用）。
  • 修正方向：用明確約束把它拉回來；縮小範圍；把行為規範寫進 CLAUDE.md。
  • 工程向：測試「寫—跑—修」循環、TDD、補覆蓋率、跨語言移植、效能優化到指定指標、資安稽核。

④ Ship — 交付與發布

🎯 PM／行銷／文件團隊適用
  • 從 diff 與歷史自動生成 commit message、release notes——不用手動描述做了什麼。
  • 從 issue 開 PR、建 CI 流程、解合併衝突。

⑤ Operate — 維運、除錯、分析、自動化

🎯 資料分析與自動化 PM 必看
  • 除錯：描述症狀，讓它追根因並修復（不用你先猜是哪裡壞）。
  • 事故應對：一次列出所有證據來源（日誌、部署、設定變更）交叉比對，而不是一條一條查。
  • 資料分析：直接給原始檔（呼應原則 5）、指定輸出格式（原則 6）。
  • 自動化：把重複工作變成可重用的 skill、設 hooks 自動觸發、接 MCP 工具——這就是第 7 章的自動化階梯。
SECTION 03

不同角色怎麼用

角色最該用的提示詞
PM架構問題、規格訪談、會議轉工單、邊界／空狀態盤點、資料分析
設計師改動影響評估、mockup 轉原型、設計與實作視覺比對、修視覺 bug
行銷／文件文字統一批次更新、依範本產草稿、發布前內容審查、release notes
資安／維運漏洞掃描、事故調查、日誌分析

換句話說：這個庫叫「Claude Code」提示詞庫，但有一半以上不需要你會寫程式。PM 用得最多的是 Discover（問清楚現況）、Design（把想法變規格）與 Operate（分析與自動化）。

SECTION 04

三個要先知道的限制

  1. 跨對話不會自動記得：背景脈絡會重置。要維持狀態，靠 CLAUDE.md 或記憶功能——呼應第 3 章「Cowork 要建 Project 帶脈絡」。
  2. 有些提示詞有工具依賴：需要先接好 MCP 伺服器、GitHub 整合或資料庫連線才跑得動。
  3. 效果取決於你的地基：既有測試、程式規範文件、清楚的專案慣例越完整，這些提示詞越有效。地基爛，AI 也只能跟著爛。
✓ PM 的啟示：第 3 點對產品同樣成立——你的產品文件、指標定義、決策紀錄越清楚，AI 幫得上的忙就越多。整理脈絡本身就是投資。
SECTION 05

接下來做什麼

  1. 用 Prompt 組裝器 練一次，刻意把六條原則都放進去。
  2. 去 PM Prompt 資料庫 挑一個你常用的，用六條原則改寫升級。
  3. 回頭讀 第 4 章：Prompt 與脈絡完整指南，把四元素與這六條原則對起來。
  4. 把改好、驗證過的提示詞存進團隊的 Prompt 庫（第 7 章的自動化第一階）。
